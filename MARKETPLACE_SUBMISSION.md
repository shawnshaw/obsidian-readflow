# ReadFlow Marketplace Submission Checklist

This file is the final handoff checklist for publishing ReadFlow to the
Obsidian community plugin marketplace.

## Before creating the public plugin repository

- Prepare a dedicated public GitHub repository for ReadFlow.
- Copy or publish the standalone staging folder generated by:

```bash
cd .obsidian/plugins/readflow
python3 export_standalone.py
```

- Use the exported folder under `Scripts/obsidian-plugins-release/` as the base of the standalone repo.

## Before creating a GitHub release

- Build the plugin:

```bash
npm install
npm run build
```

Or use the local portable Node flow:

```bash
bash build_with_local_node.sh
```

- Validate release files:

```bash
python3 check_release.py
```

- Confirm these files are present and up to date:
  - `manifest.json`
  - `versions.json`
  - `main.js`
  - `styles.css`
  - `README.md`
  - `LICENSE`

## GitHub release requirements

- Create a Git tag that exactly matches the plugin version, for example `0.2.2`
- Create a GitHub release for that exact tag
- Upload these release assets:
  - `manifest.json`
  - `main.js`
  - `styles.css`

## Obsidian community review expectations

- README should clearly explain what the plugin does
- README should include at least a few real screenshots before submission
- The plugin should work in a clean vault
- The plugin ID in `manifest.json` should stay stable
- Version numbers in `manifest.json`, `package.json`, and GitHub release tag must match
- `versions.json` must map plugin version to minimum supported Obsidian version

## Current known remaining tasks before public submission

- Replace screenshot recommendations in `README.md` with actual screenshots
- Move ReadFlow into its own dedicated public repository
- Smoke test in a clean Obsidian vault with a fresh plugin install using `SMOKE_TEST_CHECKLIST.md`
- Decide whether to keep `isDesktopOnly: true` as a permanent product constraint
